01 — Overview

What is Erlang-C?

Erlang-C is the industry-standard formula for call centre staffing. It tells you the probability that an incoming call will have to wait for an available agent, given a known volume of calls, average handle time, and number of agents on duty.

Named after Danish mathematician A.K. Erlang, the formula is derived from queuing theory (specifically the M/M/c queue model). It assumes:

Poisson Arrivals

Calls arrive randomly and independently, following a Poisson distribution — the standard assumption for telephony traffic.

Exponential Service

Handle times are exponentially distributed — a simplification that works well in practice for aggregate-level planning.

Infinite Queue

Callers wait indefinitely (no abandonment). This produces conservative estimates — real SL is often slightly better.

Steady State

Traffic intensity must be less than the number of agents, otherwise the queue grows without bound.

💡
Practical noteIn practice Erlang-C overestimates queue times because it ignores call abandonment. For centres with high abandonment rates, consider Erlang-A (which accounts for patience) as a supplement.
02 — Getting Started

Quick Start

Get your first results in under two minutes:

  1. Prepare your CSV file

    Export call data from your ACD/WFM system with at least a date column and a call volume column. Add an hour column if you want hourly breakdowns, and an aht column if handle time varies by period.

  2. Upload the file

    Drag your CSV onto the upload area, or click to browse. The tool reads it entirely in your browser — no data is uploaded to any server.

  3. Map your columns

    Common column names (date, hour, calls, aht) are auto-detected. Use the dropdowns to correct any that weren't matched.

  4. Set your parameters

    Enter your number of agents (N), target service level %, the answer time threshold (e.g. 80% in 20 seconds = 80 / 20), and a fixed AHT if your CSV doesn't contain one.

  5. Click Calculate

    Results appear instantly — summary stats, an interactive chart, and a full table. Switch between Daily and Hourly tabs to explore both granularities. Export to CSV when you're done.

05 — Reference

Understanding the Outputs

Each row in the results table represents one aggregated period (a day or an hour).

Traffic (Erlangs)

Traffic intensity A — the offered load measured in Erlangs. Calculated as:

A = (Calls × AHT) / Period_seconds
Calls = arrivals in the period AHT = average handle time in seconds Period_seconds = 86 400 for daily · 3 600 for hourly

One Erlang = one agent busy 100% of the time. An A of 8.5 means you need at least 9 agents just to keep up — before any queue buffer.

Utilisation %

Agent occupancy: A / N × 100. Values above ~85% are a warning sign — even small traffic spikes will cause long queues. Values above 100% mean the centre is mathematically overloaded.

Erlang-C (P[queuing])

The headline output — the probability that an arriving call will not find a free agent and must queue. A value of 0.35 means 35% of calls wait. Lower is better; above 0.5 is typically unacceptable.

Expected Wait E[W] (sec)

The average waiting time for calls that do queue, in seconds. Note this is conditional — the unconditional average wait (for all calls) is E[W] × Erlang-C.

Service Level %

Percentage of calls answered within your SL Threshold. Green = at or above target; amber = within 10% below target; red = significantly below target.

Recommended Agents

The minimum number of agents needed to achieve your Target SL % within the SL Threshold, given the traffic for that period. Compare this to your actual staffed N to identify gaps.

06 — Reference

The Erlang-C Formula

The probability that a call must wait (the Erlang-C value) is given by:

C(N, A) = [ (A^N / N!) × N/(N−A) ] / [ Σ(k=0 to N−1) A^k/k! + (A^N / N!) × N/(N−A) ]
N = number of agents A = traffic intensity (Erlangs) Valid only when A < N (stable system)

This is numerically unstable for large N due to factorial growth. The calculator uses log-factorial arithmetic to avoid overflow, computing each term as exp(k·ln(A) − ln(k!)) and summing in log space:

// Log-factorial method — stable for any N
function erlangC(N, A) {
  if (A >= N) return 1; // overloaded
  let sumTerms = 0, logFactK = 0;
  for (let k = 0; k < N; k++) {
    if (k > 0) logFactK += Math.log(k);
    sumTerms += Math.exp(k * Math.log(A) - logFactK);
  }
  let logFactN = logFactK + Math.log(N);
  let lastTerm = Math.exp(N * Math.log(A) - logFactN);
  let erlangB  = lastTerm / (sumTerms + lastTerm);
  return erlangB / (1 - (A / N) * (1 - erlangB));
}
07 — Reference

Service Level Calculation

Given the Erlang-C value C, the probability a call is answered within T seconds is:

SL(T) = 1 − C(N,A) × e^( −(N−A) × T / AHT )
T = SL threshold in seconds (your "within X seconds" target) AHT = average handle time in seconds C(N,A) = Erlang-C value for this period

The Recommended Agents figure iterates N upward from ceil(A)+1 until this formula returns ≥ your Target SL %.

✅
Common service standard: 80/20The industry benchmark "80% of calls answered within 20 seconds" means Target SL = 80% and SL Threshold = 20 sec. Set both in the Parameters panel.
08 — Guides

Daily vs Hourly Analysis

The calculator produces two result sets from the same CSV:

Daily results

All rows for the same date are summed (calls) and averaged (AHT). The period length used is 86 400 seconds. This gives you a day-level view of traffic trends — useful for spotting day-of-week patterns or seasonal shifts.

Hourly results

Available only when your CSV includes an hour column. Rows with the same hour value are summed across all dates in the file. The period length is 3 600 seconds. This gives you an intraday staffing curve — typically the most actionable output for scheduling.

⚠️
Agents (N) applies to all periodsThe tool uses one agent count for every row. For a more precise analysis, export the results CSV and run scenarios by adjusting N for different time slots.

Reading the chart

The chart overlays three metrics on two axes: Erlang-C (left axis, 0–1) as a filled line, and both Service Level % and Utilisation % (right axis, 0–100%) as a line and bars respectively. Peaks in Erlang-C that coincide with dips in Service Level show your highest-pressure periods.

Recommended Agents vs actual N

If Recommended Agents consistently exceeds your input N, you are understaffed for that target. If it is consistently lower, you may have headroom to reduce staffing or improve SL by redeploying agents to peak hours.

10 — Guides

Known Limitations

  • No abandonment modelling Erlang-C assumes callers wait forever. Real queues have abandonment, which reduces load. This makes Erlang-C a conservative (safe) estimate.
  • Single agent group The formula assumes a homogeneous agent pool with equal skill. Skills-based routing requires a different model.
  • Steady-state assumption The model treats each period as a steady state. Short intervals (15 min) with rapidly changing traffic are less accurate than hourly or daily aggregations.
  • One N for all periods The calculator applies a single agent count to every row. Export results to model variable staffing levels.
11 — FAQ

Frequently Asked Questions

Why is Expected Wait showing ∞?

This happens when Traffic (Erlangs) ≥ Agents (N). The system is overloaded and the queue will grow without bound. Increase agents or reduce traffic (AHT or call volume).

My Erlang-C is 1.0000 for every row

This means every row is in an overloaded state. Check that your AHT is in seconds (not minutes), and that your agent count N is realistic for your data.

The hourly tab shows no data

Map the Hour Column in step 02 of the sidebar. Without it, hourly aggregation is skipped.

Can I model multiple agent groups?

Not directly — Erlang-C is a single-queue model. Run the calculator separately for each skill group and combine results manually.

Is my data stored anywhere?

No. All processing happens entirely in your browser using JavaScript. Nothing is sent to a server.

How do I convert minutes to seconds for AHT?

Multiply by 60. An AHT of 5.5 minutes = 330 seconds. Either pre-process your CSV or enter the fixed override (in seconds) in the Parameters panel.
